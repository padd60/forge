# Architecture

## Overview

forge is the composition of two systems:

1. **A three-layer rule system** where each module contributes mechanical ESLint rules, Claude Code skills, and/or Evaluator rubrics.
2. **A Planner-Generator-Evaluator (P-G-E) pipeline** that orchestrates AI code generation with physical context isolation and file-based handoff.

Together they guarantee that AI-generated code either conforms to the architecture the user chose or is rejected with a specific, scored explanation of why.

---

## Three-layer rule system

| Layer | Format | Enforcement | Where it runs |
|---|---|---|---|
| **Mechanical** | ESLint rules + tsc config | **Block** — fails pre-commit and CI | `eslint.config.js` generated by `forge init` |
| **Skill** | Markdown with YAML frontmatter | **Auto-activation** — Claude Code reads `.claude/skills/` | Real-time, while the user writes code |
| **Rubric** | Markdown + JSON scoring criteria | **Advisory** — Evaluator scores 0–10 | `/forge-eval` or `Harness.run()` |

A single module can contribute to all three layers or to just one. For example, `module-clean-code` ships 5 ESLint rules (mechanical), 3 skill files (prompt guidance), and a rubric covering SRP, intent clarity, and props drilling depth.

Each layer is defined by an optional method on the `Module` interface (`packages/core/src/module.ts`):

```ts
interface Module {
  manifest: ModuleManifest;
  eslintConfig?(): EslintConfigFragment;  // Layer 1
  skills?(): readonly SkillDef[];          // Layer 2
  rubrics?(): readonly Rubric[];           // Layer 3
}
```

## P-G-E pipeline

```
User request
    │
    ▼
┌──────────┐   spec.json    ┌───────────┐   diff.patch    ┌───────────┐
│  Planner │ ─────────────→ │ Generator │ ─────────────→ │ Evaluator │
│ (fresh)  │   spec.md      │ (sprints) │   self-check    │ (FRESH!)  │
└──────────┘                └───────────┘                └─────┬─────┘
                                  ▲                           │
                                  │    report.json            │
                                  │    (passed: false)        │
                                  └───────────────────────────┘
                                       fix loop (max 3)
```

### Key invariants

1. **The Evaluator always runs in a fresh context.** It has zero access to Generator prompts, intermediate reasoning, or tool calls. It receives only the git diff, the original spec, and the active rubrics. This is the core Anthropic insight: agents that evaluate their own work confidently praise mediocrity.

2. **Every state transition writes to disk.** The run directory (`.forge/runs/<runId>/`) contains the complete audit trail: `request.json`, `planner/spec.json`, `generator/sprint-*/diff.patch`, `evaluator/iteration-*/report.json`. Any crash is recoverable.

3. **`Harness.run()` is the programmatic mirror.** The production execution path is the Claude Code slash commands (`/forge-plan`, `/forge-generate`, `/forge-eval`), each of which spawns a sub-agent via the Task tool. `Harness.run()` in `packages/core/src/harness.run.ts` replicates the same state machine using `MockRuntime` + fixture files, enabling deterministic unit tests without LLM calls.

### State machine (`packages/core/src/harness.run.ts`)

```
1. runId, paths = computeRunPaths(forgeDir, runId)
2. io.writeRequest(paths, request)
3. spec = planner.plan(input, runtime)       // spawn sub-agent
4. io.writeSpec(paths, spec, renderSpecMd)
5. io.writePlannerHandoff(paths, handoff)
6. genResult = generator.generate(input, runtime)  // sprint loop
7. for iteration = 1..maxIterations:
     report = evaluator.evaluate(input, runtime)   // FRESH context
     io.writeEvalReport(iterDir, report)
     if report.passed → io.writeFinalReport → return
     if iteration == max → throw RunLimitError
     io.writeEvalToGeneratorHandoff → re-enter generator
```

## File-based handoff contract

```
.forge/runs/<runId>/
├── request.json                 # RunRequest (Zod-validated)
├── planner/
│   ├── spec.json                # Spec — structured plan
│   ├── spec.md                  # Human-readable spec
│   └── handoff.json             # PlannerToGenerator handoff
├── generator/
│   ├── sprint-01/
│   │   ├── plan.md
│   │   ├── diff.patch
│   │   ├── self-check.json      # ESLint/tsc results
│   │   └── handoff.json
│   └── sprint-02/...
└── evaluator/
    ├── iteration-01/
    │   ├── report.json          # EvalReport { score, passed, violations }
    │   ├── report.md
    │   └── handoff.json         # If fix loop → back to generator
    └── final.json               # Last EvalReport (passed or failed)
```

All schemas are defined in `packages/schemas/src/` and validated at read/write boundaries by `packages/core/src/run-io.ts`.

## Module system

### Precedence and conflict resolution

Each module declares a `precedence: number` in its manifest. When two modules set the same ESLint rule to different values, `packages/core/src/conflict-resolver.ts` picks the stricter one (lower precedence number wins) and logs the conflict to `.forge/conflicts.log`.

### `loadModules()` (`packages/core/src/module-loader.ts`)

Takes a registry of all available modules and a user's `activeModules` list. Returns `{ active, skipped }` where `skipped` entries include a reason (missing dependency, incompatible version, etc.). Modules are sorted by precedence for deterministic ESLint config generation.

## Enforcement levels

| Level | Mechanical (pre-commit) | Evaluator |
|---|---|---|
| `hybrid` (default) | Block on violation | Advisory — reports score but doesn't block |
| `block-all` | Block | Block if `evaluator.minScore` not met |
| `advisory-only` | Warn only (no husky hook) | Advisory |

Set via `forge init --enforcement <level>` or in `.forge/config.json`.

## Extending forge

To create a new module (e.g. `module-react-query`):

1. Create `packages/module-react-query/` with the same structure as `packages/module-fsd/`.
2. Implement `eslintConfig()`, `skills()`, and/or `rubrics()` on the module's `defineModule()` call.
3. Register the module in `apps/cli/src/registry.ts`.
4. Add ESLint rules to `packages/eslint-plugin-forge/src/rules/` if needed.
5. Run `pnpm seed:examples` and `pnpm -r test` to verify.
